## Key Technical Decisions

### Why WebSocket + SSE (Not Just WebSocket)?

**WebSocket** (bidirectional):
- Best for: Interactive controls, task submission, progress updates
- Requires: Active connection management, reconnection logic
- Use case: User starts RG flow â†’ server sends progress every 100ms

**SSE** (unidirectional serverâ†’client):
- Best for: Log streaming, one-way data push
- Simpler: Automatic reconnection, standard HTTP
- Use case: Transparency engine emits logs continuously

**Decision:** Use **both** - WebSocket for control plane, SSE for data plane

### Why Three.js (Not Plotly/Matplotlib)?

**Three.js**:
- Native WebGL performance (60 FPS)
- Full control over 3D scene
- GPU-accelerated particles for field visualization
- Custom shaders for phase coloring

**Plotly/Matplotlib**:
- Static or low-frame-rate
- Limited customization
- Cannot handle large particle systems

**Decision:** Three.js for 3D, Recharts for 2D plots

### Why Zustand (Not Redux)?

**Zustand**:
- Minimal boilerplate (100 lines vs 500)
- No context provider wrapping
- Better TypeScript support
- Easier to learn

**Redux Toolkit**:
- More verbose
- Requires redux-thunk/saga for async
- Larger bundle size

**Decision:** Zustand for simplicity, scales to medium apps
